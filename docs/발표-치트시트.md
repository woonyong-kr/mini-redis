# 발표 치트시트 — mini-redis 라이브 데모

## 시작 전 준비

터미널 2개를 열어 나란히 배치합니다.

**터미널 A — 서버 실행**

```bash
cd mini-redis
rm -f data/appendonly.aof data/dump.rdb
MINI_REDIS_APPENDONLY=yes \
MINI_REDIS_RDB_ENABLED=yes \
MINI_REDIS_RDB_SAVE_INTERVAL_SECONDS=5 \
python3 server.py --host 0.0.0.0 --port 6399
```

**터미널 B — redis-cli 접속**

```bash
cd mini-redis
./redis-cli -p 6399
```

> `redis-cli` 바이너리가 프로젝트 폴더 안에 있습니다.

> 말할 것: "오늘 데모에서는 저장/복구 시나리오를 보기 위해 AOF와 RDB를 모두 켜고 시작합니다. 복구는 AOF 우선, 없으면 RDB 순서입니다."

## 순서 3 — TTL 자동 만료 (40초)

```
127.0.0.1:6399> SET session:abc "token123" EX 5
OK

127.0.0.1:6399> TTL session:abc
(integer) 4

127.0.0.1:6399> EXISTS session:abc
(integer) 1
```

> 여기서 5초 기다린 후:

```
127.0.0.1:6399> TTL session:abc
(integer) -2

127.0.0.1:6399> GET session:abc
(nil)
```

> 말할 것: "TTL 만료는 Lazy + Active 이중 전략으로 동작합니다. Lazy는 조회 시점에 확인해서 삭제하고, Active는 100ms마다 백그라운드에서 랜덤 샘플링으로 만료 키를 청소합니다. 샘플에서 만료 비율이 25% 이상이면 같은 주기에 추가 패스를 최대 4회까지 수행합니다."

## 예상 질문 & 답변

**Q. 실제 Redis와 성능 차이는?**

Python 인터프리터 오버헤드로 단순 SET/GET 기준 실제 Redis 대비 약 30~50% 느립니다. 학습 목적이므로 성능보다 구조 재현에 집중했습니다. uvloop 덕분에 순수 asyncio 대비 2~4배 빠릅니다.

**Q. 왜 asyncio + uvloop인가요?**

Redis는 싱글 스레드 + I/O 멀티플렉싱 (epoll/kqueue) 구조입니다. asyncio가 이 구조를 Python에서 재현하는 가장 자연스러운 방법이고, uvloop는 C로 작성된 libuv 이벤트 루프를 사용해 기본 asyncio보다 2~4배 빠릅니다.

**Q. Hash의 promote는 언제 일어나나요?**

필드 수가 32개를 초과하거나, 키/값 중 하나가 64바이트를 초과할 때 compact list에서 ChainedHashTable로 자동 전환됩니다. 실제 Redis의 listpack → hashtable 전략과 동일합니다.

**Q. TTL은 어떻게 관리하나요?**

Lazy expiry와 Active expiry를 함께 씁니다. Lazy는 GET 같은 읽기 시 만료를 체크하고, Active는 100ms마다 랜덤 20개를 샘플링해서 만료 키를 삭제합니다. 만료 비율이 25%를 넘으면 재샘플링(최대 4회)합니다.

**Q. 저장 복구는 어떻게 시연하나요?**

발표에서는 AOF를 메인으로 보여주면 됩니다. 서버를 `MINI_REDIS_APPENDONLY=yes`로 실행하고 데이터를 저장한 뒤 서버를 재시작하면, AOF replay로 같은 상태가 복구됩니다. RDB는 함께 켜서 "주기적 스냅샷도 있다"는 보조 설명으로 붙이면 됩니다.

**Q. 왜 SkipList를 직접 구현했나요?**

Redis의 Sorted Set이 SkipList를 사용하기 때문입니다. Python의 sorted container 라이브러리를 쓸 수도 있지만, 직접 구현해서 span 배열을 통한 rank 계산과 레벨 구조를 이해하는 것이 목적이었습니다.

**Q. Eviction은 어떻게 동작하나요?**

`maxmemory`를 설정하면 쓰기 연산마다 메모리 사용량을 체크합니다. 초과 시 정책에 따라 키를 삭제합니다. 만약 eviction으로도 공간이 부족하면 현재 쓰기를 롤백하고 OOM 에러를 반환합니다.
