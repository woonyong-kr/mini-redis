# 발표 치트시트 — mini-redis 라이브 데모

## 시작 전 준비

터미널 2개를 열어 나란히 배치합니다.

**터미널 A — 서버 실행**

```bash
cd mini-redis
python3 server.py --port 6399
```

**터미널 B — redis-cli 접속**

```bash
cd mini-redis
./redis-cli -p 6399
```

> `redis-cli` 바이너리가 프로젝트 폴더 안에 있습니다.

## 순서 3 — TTL 자동 만료 (40초)

```
127.0.0.1:6399> SET session:abc "token123" EX 5
OK

127.0.0.1:6399> TTL session:abc
(integer) 4

127.0.0.1:6399> EXISTS session:abc
(integer) 1
```

> 여기서 5초 기다린 후:

```
127.0.0.1:6399> TTL session:abc
(integer) -2

127.0.0.1:6399> GET session:abc
(nil)
```

> 말할 것: "TTL 만료는 Lazy + Active 이중 전략으로 동작합니다. Lazy는 조회 시점에 확인해서 삭제하고, Active는 100ms마다 백그라운드에서 랜덤 샘플링으로 만료 키를 청소합니다."

## 순서 4 — Hash (30초)

> **주의**: `user:1`은 이미 String으로 사용 중 → `profile:1` 키로 테스트

```
127.0.0.1:6399> HSET profile:1 name "Alice" age "28" city "Seoul"
(integer) 3

127.0.0.1:6399> HGET profile:1 name
"Alice"

127.0.0.1:6399> HGETALL profile:1
1) "name"
2) "Alice"
3) "age"
4) "28"
5) "city"
6) "Seoul"

127.0.0.1:6399> HLEN profile:1
(integer) 3
```

> 말할 것: "Hash는 필드가 32개 이하면 compact list로 메모리를 아끼고, 초과하면 MurmurHash3 기반 OpenAddressHashTable로 자동 승격합니다."

## 예상 질문 & 답변

**Q. 실제 Redis와 성능 차이는?**

Python 인터프리터 오버헤드가 있어서 단순 SET/GET은 실제 Redis 대비 5~10배 느립니다. 학습 목적이므로 성능보다 구조 재현에 집중했습니다.

**Q. 왜 asyncio + uvloop인가요?**

Redis는 싱글 스레드 + I/O 멀티플렉싱 구조입니다. asyncio가 이 구조를 Python에서 재현하는 가장 자연스러운 방법이고, uvloop는 C로 작성된 libuv 이벤트 루프를 사용해 기본 asyncio보다 2~4배 빠릅니다.

**Q. Hash의 promote는 언제 일어나나요?**

필드 수가 32개를 초과하거나, 키/값 중 하나가 64바이트를 초과할 때 compact list에서 OpenAddressHashTable로 자동 전환됩니다. 실제 Redis의 listpack → hashtable 전략과 동일합니다.

**Q. TTL은 어떻게 관리하나요?**

Lazy expiry와 Active expiry를 함께 씁니다. Lazy는 GET 같은 읽기 시 만료를 체크하고, Active는 100ms마다 랜덤 20개를 샘플링해서 만료 키를 삭제합니다. 만료 비율이 25%를 넘으면 재샘플링(최대 4회)합니다.
