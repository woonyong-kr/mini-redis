# mini-redis — Python으로 Redis 직접 구현하기

> Redis의 핵심 동작 원리를 Python으로 재현한 학습 프로젝트

---

## 프로젝트 목적

Redis를 단순히 사용하는 것이 아니라, 내부 구조를 직접 구현함으로써 동작 원리를 체득하는 것이 목표입니다. 실제 Redis 소스코드가 선택한 자료구조, 알고리즘, 프로토콜을 동일한 방식으로 Python으로 재구현했습니다.

---

## 전체 아키텍처

```
클라이언트 (redis-cli / nc)
        │  RESP Protocol
        ▼
  TCP Server (asyncio + uvloop)
        │
        ▼
  Command Dispatcher
  ┌─────┬─────┬──────┬─────┬──────┐
  │ str │hash │ list │ set │ zset │
  └─────┴─────┴──────┴─────┴──────┘
        │
        ▼
  DataStore + ExpiryManager
        │
        ▼
  PersistenceManager (AOF / RDB)
```

## 1. 비동기 서버 — asyncio + uvloop

### asyncio란?

Python 표준 비동기 I/O 프레임워크입니다. 싱글 스레드에서 이벤트 루프를 사용해 여러 클라이언트를 동시에 처리합니다. `await` 키워드로 I/O 대기 중에 다른 작업을 처리할 수 있어 수천 개의 동시 연결을 효율적으로 다룰 수 있습니다.

### uvloop를 사용한 이유

uvloop는 asyncio의 이벤트 루프를 **libuv(Node.js의 비동기 엔진)**로 교체하는 라이브러리입니다. 기본 asyncio 루프보다 **2~4배 빠른 처리 속도**를 제공합니다.

```
기본 asyncio
Python → asyncio event loop → OS I/O

uvloop 사용 시
Python → libuv event loop → OS I/O
                ↑
         C로 작성된 고성능 엔진
```

실제 Redis도 싱글 스레드 + I/O 멀티플렉싱(epoll/kqueue) 구조입니다. uvloop를 사용하면 이 구조와 가장 유사한 환경을 Python에서 만들 수 있습니다.

```python
# server.py에서의 적용
import uvloop

uvloop.install()  # asyncio의 기본 루프를 uvloop로 교체
asyncio.run(main())
```

### asyncio + uvloop 조합의 핵심

클라이언트가 명령을 보내면 서버는 I/O를 기다리는 동안 다른 클라이언트의 요청을 처리합니다. CPU 작업(명령 실행)은 빠르게 끝나고 대부분의 시간을 I/O 대기에 쓰는 Redis 특성상, 비동기 단일 스레드 구조가 적합합니다.

## 2. RESP 프로토콜

Redis 클라이언트와 서버가 통신하는 텍스트 기반 직렬화 프로토콜을 직접 구현했습니다.

```
클라이언트가 보내는 형식 (배열 + 벌크 문자열):
*3\r\n         ← 인자 3개
$3\r\n         ← 3바이트
SET\r\n
$5\r\n         ← 5바이트
mykey\r\n
$5\r\n         ← 5바이트
hello\r\n

서버 응답 형식:
+OK\r\n        ← Simple String (성공)
-ERR msg\r\n   ← Error
:42\r\n        ← Integer
$5\r\nhello\r\n ← Bulk String
*3\r\n...      ← Array
```

파이프라이닝도 지원합니다. 하나의 TCP 연결에 여러 명령을 연속으로 전송하면 순서대로 처리하고 응답합니다.

## 3. Hash 자료구조 — listpack → hashtable 전략

### Redis가 Hash를 두 가지로 관리하는 이유

필드가 적은 Hash에 즉시 hashtable을 쓰면 포인터와 구조체 오버헤드로 메모리 낭비가 발생합니다. Redis는 작을 땐 compact 구조로, 커지면 hashtable로 자동 전환합니다. 이 프로젝트도 동일하게 재현했습니다.

### Compact 구조 (listpack)

```
필드 수 ≤ 32개  &&  각 값 크기 ≤ 64 bytes

내부 표현:
[("name", "Alice"), ("age", "30"), ("email", "alice@example.com")]
→ 순서가 있는 리스트로 메모리에 연속 저장
```

### OpenAddressHashTable (조건 초과 시 자동 promote)

```
필드 수 > 32개  또는  값 크기 > 64 bytes

Hash 알고리즘: MurmurHash3-32 (x86 버전 직접 구현)
충돌 해결:     Double Hashing (오픈 어드레싱)
슬롯 상태:     EMPTY / OCCUPIED / TOMBSTONE
```

### MurmurHash3-32 — 직접 구현

```python
def murmurhash3_32(value, seed=0):
    # 4바이트씩 블록으로 나눠 처리
    for block in blocks:
        k1 = (block * c1) & 0xFFFFFFFF
        k1 = rotate_left(k1, 15)
        k1 = (k1 * c2) & 0xFFFFFFFF
        h1 ^= k1
        h1 = rotate_left(h1, 13)
        h1 = (h1 * 5 + 0xE6546B64) & 0xFFFFFFFF
    # 나머지 바이트 처리 + 파이널 믹싱
    ...
```

MurmurHash3는 분산이 균등하고 충돌이 적어 Redis, Kafka, Cassandra 등 여러 프로젝트에서 사용하는 해시 함수입니다.

### Double Hashing

```
첫 번째 해시:  h1(key) = MurmurHash3(key) % capacity
두 번째 해시:  step(key) = ((h1 >> 16 ^ h1 << 1) | 1) % capacity

탐색 순서: (h1 + i * step) % capacity  (i = 0, 1, 2, ...)
```

step을 항상 홀수(| 1)로 만들어 capacity가 2의 거듭제곱이더라도 전체 슬롯을 순회할 수 있도록 보장합니다.

### 동적 리사이징

```
load factor > 0.7  →  capacity × 2  (grow)
load factor < 0.2  →  capacity / 2  (shrink)
```

삭제 시 슬롯을 TOMBSTONE으로 표시해 탐색 체인을 끊지 않고, 리사이징 시에는 TOMBSTONE을 제외하고 재삽입합니다.

### Promote 흐름 요약

```
HSET user name "Alice"
    → compact list: [("name", "Alice")]   ← 소형 (직렬 탐색)

HSET user ... (33번째 필드 추가)
    → _should_promote() → True
    → OpenAddressHashTable 생성
    → compact 데이터 전부 이전 (O(n) 1회)
    → 이후 모든 연산은 O(1) 해시 탐색
```

## 4. TTL 만료 전략 — Lazy + Active

Redis는 만료 키 삭제를 두 가지 방식으로 처리합니다. 이 프로젝트도 동일하게 구현했습니다.

### Lazy Expiry (지연 삭제)

키에 접근할 때(`GET`, `TTL`, `EXISTS` 등) 만료 여부를 체크하고, 만료됐으면 그 자리에서 삭제합니다.

```python
def get_ttl(self, key: str) -> float:
    remaining = self._expiry[key] - time.time()
    if remaining <= 0:
        self._store.delete(key, reason="expiry")  # ← 접근 시점에 삭제
        return -2
    return remaining
```

**장점**: 오버헤드가 없음
**단점**: 아무도 조회하지 않는 만료 키는 메모리에 계속 남아있음

### Active Expiry (능동 샘플링 삭제)

**메모리가 많아서 삭제하는 게 아닙니다.** 100ms마다 백그라운드에서 랜덤으로 키를 샘플링해서 만료된 키를 능동적으로 삭제합니다.

```
매 100ms마다:
  1. TTL이 설정된 키 중 랜덤 20개 샘플링
  2. 만료된 키 모두 삭제
  3. 샘플 중 만료 비율 > 25% 이면 재샘플링 (최대 4회 반복)
```

```python
async def active_expiry_loop(self) -> None:
    while True:
        self.evict_expired_samples()     # ← 샘플 기반 능동 삭제
        await asyncio.sleep(0.1)         # ← 100ms 대기
```

**장점**: 아무도 조회하지 않아도 만료 키가 정리됨
**단점**: 전체 키를 스캔하지 않으므로 일부 만료 키가 잠시 남을 수 있음 (허용 범위)

### Lazy + Active 조합의 효과

```
조회 있음  →  Lazy가 즉시 삭제
조회 없음  →  Active가 100ms 이내에 샘플로 삭제
```

전체 키를 매번 스캔하는 것보다 훨씬 적은 오버헤드로 만료 키를 관리할 수 있습니다. 실제 Redis도 이 전략을 사용합니다.

## 5. Sorted Set — SkipList + dict

ZSet은 점수(score) 기반 정렬 + O(1) 점수 조회를 동시에 지원해야 합니다.

```
dict:      { "Alice": 100, "Bob": 200 }  ← O(1) 점수 조회
SkipList:  [Dave(50) → Alice(100) → Charlie(150) → Bob(200)]  ← O(log n) 범위 조회
```

SkipList는 각 노드에 span(건너뛴 노드 수)을 저장해 rank 계산을 O(log n)에 처리합니다.

## 6. 영속성 — AOF / RDB

### AOF (Append-Only File)

쓰기 명령을 RESP 형식 그대로 파일에 append합니다. 재시작 시 파일의 명령을 순서대로 재실행해서 데이터를 복원합니다.

```
fsync 정책:
  always   → 명령마다 fsync (가장 안전, 느림)
  everysec → 1초마다 fsync  (기본값, 균형)
  no       → OS에 위임        (빠름, 덜 안전)
```

### RDB (Snapshot)

일정 주기로 현재 데이터를 RESP 명령 스트림으로 직렬화해 파일에 저장합니다. `.tmp` 파일에 먼저 쓴 뒤 atomic rename으로 교체해 파일 손상을 방지합니다.

```
RDB 파일 구조:
MINIRDB1\r\n   ← 매직 헤더 (8바이트)
RESP 명령들    ← SET key val / HSET key f v / ZADD key score member ...
PEXPIREAT key ms   ← TTL 재설정
```

## 7. 메모리 제한 정책

`maxmemory` 초과 시 4가지 정책 중 하나로 동작합니다.

| 정책 | 동작 |
|---|---|
| `noeviction` | OOM 에러 반환, 쓰기 거부 |
| `allkeys-random` | 전체 키 중 랜덤 삭제 |
| `allkeys-lru` | 가장 오래 사용하지 않은 키 삭제 |
| `volatile-ttl` | TTL이 가장 짧게 남은 키 삭제 |

## 10. 학습 포인트 정리

이 프로젝트를 통해 체득한 핵심 개념들입니다.

**프로토콜 설계**: RESP처럼 단순한 텍스트 프로토콜이 어떻게 고성능을 달성하는지 이해했습니다.

**자료구조 선택의 이유**: 엔트리가 적을 때 listpack을 쓰는 것이 단순한 최적화가 아니라 캐시 지역성까지 고려한 설계임을 알았습니다.

**해시 함수 구현**: MurmurHash3를 직접 구현하면서 분산 균일성, 비트 믹싱의 원리를 이해했습니다.

**만료 전략의 트레이드오프**: Lazy만으로는 메모리 누수, Active만으로는 과도한 CPU 사용 — 두 전략의 조합이 왜 필요한지 코드로 직접 확인했습니다.

**비동기 I/O**: asyncio + uvloop 구조가 멀티스레드 없이도 수천 연결을 처리할 수 있는 이유를 이해했습니다.
