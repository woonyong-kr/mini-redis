# mini-redis 발표 원고

## 1. 프로젝트 소개

안녕하세요.  
이번 프로젝트는 Redis를 단순히 사용하는 것이 아니라, Redis의 핵심 구조를 Python으로 직접 구현해보는 mini-redis 프로젝트입니다.

저희 목표는 두 가지였습니다.

1. 기존 Redis 클라이언트가 그대로 붙을 수 있도록 RESP 기반 서버를 직접 만든다.
2. 공식 Redis, mini-redis, MongoDB를 같은 시나리오로 비교해서 메모리 기반 서버와 일반 DB의 차이를 데이터로 보여준다.

---

## 2. 전체 구조

전체 흐름은 다음 5단계입니다.

```
Client ──TCP──> server.py ──> parser.py ──> dispatcher.py ──> *_cmds.py
                                                                 │
                                                                 ▼
                                                            DataStore
                                                           (store 계층)
                                                                 │
                                    encoder.py <─────────────────┘
                                        │
                                   Client에 응답
```

1. **TCP 연결 수신** — `asyncio.start_server()`로 소켓을 열고, 연결마다 코루틴 생성
2. **RESP 파싱** — `parser.py`가 바이트 스트림을 `["SET", "key", "value"]` 형태로 변환
3. **명령 디스패치** — `dispatcher.py`가 명령 이름으로 핸들러 함수를 탐색
4. **저장 계층 처리** — 핸들러가 `DataStore`의 메서드를 호출해 데이터 조작
5. **RESP 응답 반환** — `encoder.py`가 결과를 RESP 바이트로 인코딩해서 전송

---

## 3. 서버 동작 방식

이 서버는 멀티 스레드 서버가 아니라, **단일 이벤트 루프 기반 비동기 서버**입니다.

프로그램이 시작되면 `Server()`를 만들고 `asyncio.run(server.start())`를 호출합니다.

그 다음:

1. TTL 청소용 백그라운드 작업을 등록합니다.
2. `asyncio.start_server()`로 TCP 소켓을 엽니다.
3. 이벤트 루프가 I/O 이벤트를 기다립니다.
4. 클라이언트가 접속하면 연결마다 `handle_client(reader, writer)` 코루틴이 생성됩니다.
5. 각 연결 코루틴은 `while True`로 요청을 계속 읽고 처리합니다.

여기서 중요한 점은,  
`await reader.read()`나 `await writer.drain()`을 만날 때 해당 코루틴이 잠시 대기 상태가 되고,  
그 사이 이벤트 루프가 다른 연결이나 TTL 정리 작업을 실행한다는 점입니다.

즉, **하나의 스레드에서 여러 연결을 번갈아 처리하는 구조**입니다.

### 느린 클라이언트 보호

한 클라이언트가 서버 전체를 느리게 만들지 않도록 5가지 보호 장치를 넣었습니다.

| 보호 장치 | 기본값 | 동작 |
|----------|--------|------|
| Idle timeout | 30초 | 유휴 연결 자동 종료 |
| Write drain timeout | 5초 | 응답을 못 받는 느린 클라이언트 차단 |
| Max input buffer | 1MB | 과도한 요청 크기 제한 |
| Max output buffer | 256KB | 응답 버퍼 메모리 보호 |
| Max commands/tick | 128개 | 한 연결이 이벤트 루프 독점 방지 |

## 4. RESP 프로토콜

Redis 클라이언트와 통신하려면 RESP 프로토콜을 이해해야 합니다.

예를 들어 `SET mykey hello`는 다음과 같이 직렬화됩니다:

```
*3\r\n$3\r\nSET\r\n$5\r\nmykey\r\n$5\r\nhello\r\n
```

서버는 이 바이트 스트림을 파싱해서 `["SET", "mykey", "hello"]`로 바꾸고,  
실행 결과를 다시 RESP 응답으로 인코딩해서 돌려줍니다.

응답 타입은 5가지입니다:

| prefix | 타입 | 예시 |
|--------|------|------|
| `+` | Simple String | `+OK\r\n` |
| `-` | Error | `-ERR unknown command\r\n` |
| `:` | Integer | `:42\r\n` |
| `$` | Bulk String | `$5\r\nhello\r\n` |
| `*` | Array | `*2\r\n$3\r\nfoo\r\n$3\r\nbar\r\n` |

이 덕분에 `redis-cli`, `redis-py` 같은 기존 클라이언트를 **수정 없이 그대로 사용**할 수 있습니다.

## 5. 구현한 자료구조

### String (9개 명령)

내부적으로 `bytes`로 저장합니다.  
정수로 해석 가능한 문자열은 `ENC_INT` 메타데이터를 붙여서  
`INCR`, `DECR`, `INCRBY` 같은 **카운터 패턴**에 사용합니다.

`MSET`은 도중 실패 시 모든 키를 이전 상태로 **롤백**합니다.  
이를 위해 변경 전에 `_snapshot_key()`로 스냅샷을 찍고, 실패 시 `_restore_key_snapshot()`으로 복원합니다.

### Hash (10개 명령)

Redis의 **listpack → hashtable 전략**을 재현했습니다:

1. 필드 수 ≤ 32개 & 값 크기 ≤ 64바이트 → **compact list** (튜플 리스트)
2. 임계치 초과 시 → **ChainedHashTable**로 자동 승격

해시 함수는 **MurmurHash3 x86 32-bit**을 직접 구현했습니다.

추가로 **OpenAddressHashTable** (Double Hashing + Tombstone)도 구현해서  
probing, tombstone, resize 같은 해시 테이블 동작 자체를 검증할 수 있게 했습니다.

> "작을 때는 메모리를 아끼고, 커지면 해시 테이블로 승격하는 전략을 구현했습니다."

### List (8개 명령)

`collections.deque` 기반입니다.  
양쪽 push/pop이 O(1)이라 **큐(Queue) 패턴**에 적합합니다.

### Set (8개 명령)

Python `set` 기반으로, membership 확인과 **집합 연산** (교집합, 합집합, 차집합)을 지원합니다.

### Sorted Set (8개 명령)

두 구조를 함께 사용합니다:

- `dict(member → score)` — O(1) 점수 조회
- **SkipList** — O(log n) 범위 조회 · 순위 조회

SkipList는 직접 구현했습니다:
- 최대 16레벨, 승격 확률 P=0.25
- `span` 배열로 rank 계산
- insert, delete, rank 모두 O(log n)

> "ZSet은 SkipList와 dict를 조합해서 O(log n) 정렬 + O(1) 점수 조회를 동시에 처리합니다."

## 6. 명령 디스패처 구조

`dispatcher.py`는 53개 명령을 **그룹별로 정리한 라우팅 테이블**을 구축합니다.

```python
COMMAND_TABLE = {
    # core_string
    "GET": cmd_get, "SET": cmd_set, "MGET": cmd_mget, "MSET": cmd_mset,
    "INCR": cmd_incr, "DECR": cmd_decr, "INCRBY": cmd_incrby,
    # extended_string
    "APPEND": cmd_append, "STRLEN": cmd_strlen,
    # core_generic
    "PING": cmd_ping, "DEL": cmd_del, "EXPIRE": cmd_expire, "TTL": cmd_ttl, ...
    # core_hash, core_list, core_set, core_zset, ...
}
```

모든 핸들러의 시그니처는 동일합니다:

```python
def cmd_xxx(store: DataStore, expiry: ExpiryManager, args: List[str]) -> Any
```

에러 처리:
- 존재하지 않는 명령 → `ERR unknown command`
- 타입 불일치 → `WRONGTYPE` 에러
- 메모리 한도 초과 → `OOM` 에러

## 7. TTL 관리

TTL은 두 가지 방식으로 관리합니다.

### Lazy Expiry

키를 읽거나 확인할 때 (`GET`, `EXISTS` 등) 만료 여부를 검사하고, 만료됐다면 즉시 삭제합니다.

### Active Expiry

백그라운드 루프가 100ms 간격으로 동작합니다:

1. TTL 키 중 20개를 랜덤 샘플링
2. 만료된 키 삭제
3. 삭제 비율이 25% 이상이면 → 같은 주기에 추가 패스 (최대 4회)

> **핵심:** TTL 키 전체를 매번 스캔하지 않습니다. 오버헤드를 줄이면서 만료 키가 메모리에 오래 남지 않도록 설계했습니다.

## 8. 영속성

### AOF (Append Only File)

쓰기 명령을 RESP 형태로 파일에 append합니다.  
서버가 재시작되면 이 명령들을 replay해서 상태를 복구합니다.

fsync 정책: `always` (매 명령) / `everysec` (1초 간격) / `no` (OS에 위임)

### RDB (Snapshot)

현재 메모리 상태를 커스텀 바이너리 스냅샷으로 저장합니다.  
RESP 명령 형태로 변환해서 저장하므로 replay 로직을 재사용합니다.

**복구 우선순위:** AOF 우선 → AOF 없으면 RDB

> "실시간 변경은 AOF로, 시점 저장은 RDB로 관리하고, 재시작 시 AOF 우선 복구합니다."

## 9. 메모리와 Eviction

`deep_getsizeof()`로 Python 객체를 재귀적으로 계측합니다.

`maxmemory`를 넘으면 eviction 정책이 작동합니다:

| 정책 | 동작 |
|------|------|
| `noeviction` | 쓰기 거부 (OOM 에러 반환) |
| `allkeys-random` | 무작위 키 1개 삭제 |
| `allkeys-lru` | 가장 오래 접근하지 않은 키 삭제 |
| `volatile-ttl` | TTL 설정된 키 중 만료 임박 키 우선 삭제 |

쓰기 연산 실행 → 메모리 한도 확인 → 초과 시 eviction 또는 롤백

## 10. 벤치마크

### 실행 방법

```bash
make run     # 전부 자동: venv → Docker 정리 → 서비스 기동 → 벤치마크
```

### 7개 비교 시나리오

| # | 시나리오 | 사용 명령 | 비교 대상 | 목적 |
|---|---------|----------|----------|------|
| 1 | KV Cache | GET / SET | Redis vs mini-redis | 기본 성능 차이 |
| 2 | Session Store | HSET / HGETALL | Redis vs mini-redis | Hash 연산 비용 |
| 3 | Rate Limiting | INCR + EXPIRE | Redis vs mini-redis | 원자적 카운터 속도 |
| 4 | Message Queue | LPUSH / RPOP | Redis vs mini-redis | 큐 처리량 |
| 5 | Leaderboard | ZADD / ZREVRANGE | Redis vs mini-redis | Sorted Set 성능 |
| 6 | Cache vs DB | 캐시 Hit/Miss | Redis+Mongo vs mini-redis+Mongo | 캐시 효과 정량화 |
| 7 | Pipeline | 배치 SET | Redis vs mini-redis | 네트워크 RTT 절감 효과 |

### 측정 지표

- **Mean / P50 / P95 / P99 레이턴시** — 성능 분포 파악
- **Throughput (ops/sec)** — 초당 처리량
- **Error rate** — 미구현 명령 또는 에러 비율

결과는 `benchmark/results/report.json`과 `report.csv`에 자동 저장됩니다.

## 11. Docker 구성

`docker-compose.yml`로 4개 서비스를 한 번에 기동합니다:

```
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ redis-official│  │  mini-redis   │  │   MongoDB    │
│  :6379       │  │   :6380      │  │   :27017     │
└──────┬───────┘  └──────┬───────┘  └──────┬───────┘
       │                  │                  │
       └──────────┬───────┴──────────────────┘
                  │
          ┌───────▼───────┐
          │  benchmark bot │
          │  (Python)      │
          └────────────────┘
```

- 모든 서비스는 `bench-net` 브릿지 네트워크에 연결
- 데이터 볼륨 없음 → 컨테이너 종료 시 데이터 소멸 (에페메럴)
- 환경 변수로 모든 설정 조절 가능

## 12. 결론

이 프로젝트는 Redis의 핵심 구조를 Python으로 재현한 mini-redis입니다.

핵심 성과:

- ✅ 기존 Redis 클라이언트와 통신 가능한 **RESP 서버** 구현
- ✅ **5가지 자료구조** 구현 (String, Hash, List, Set, ZSet)
- ✅ **커스텀 자료구조** 직접 구현 (MurmurHash3, Hash Table 2종, SkipList)
- ✅ **TTL** (Lazy + Active), **AOF**, **RDB**, **maxmemory**, **eviction** 구현
- ✅ **느린 클라이언트 보호** 장치 구현
- ✅ 공식 Redis · MongoDB와 **7개 시나리오 벤치마크** 구성
- ✅ `make run` **원커맨드** Docker 실행 환경

한 줄 요약:

> **"Redis를 단순히 사용하는 수준이 아니라, 내부 구조와 동작 원리를 직접 구현하고 비교할 수 있게 만든 프로젝트"**
